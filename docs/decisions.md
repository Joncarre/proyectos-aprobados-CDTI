# Registro de decisiones técnicas

Decisiones estructurales del proyecto, en orden cronológico. Las decisiones de cada fase se
añaden aquí al cerrarla.

## FASE 0 (2026-06-12)

| #   | Decisión                                               | Alternativas consideradas        | Motivo                                                                                                                                                                                                                                                     |
| --- | ------------------------------------------------------ | -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **DuckDB** como base de datos                          | SQLite, PostgreSQL               | Motor columnar vectorizado: agregaciones con filtros combinados sobre cientos de miles de filas en milisegundos. Embebido y sin operaciones. Modo `READ_ONLY` para la API; la ingesta es el único escritor y corre por CLI. Detalle completo en el README. |
| 2   | **Node + Fastify (TypeScript)** para backend e ingesta | Python + FastAPI                 | Un solo lenguaje en el monorepo; tipos compartidos backend/frontend en `shared/`; plugins oficiales de Fastify para CORS, helmet y rate-limit; cliente oficial DuckDB para Node. Confirmado por el usuario.                                                |
| 3   | **npm workspaces** como monorepo                       | pnpm, turborepo, repos separados | Cero herramientas adicionales; un `npm install`; suficiente para 4 workspaces.                                                                                                                                                                             |
| 4   | JSON crudos **fuera de git** (`data/raw/` gitignorado) | Versionarlos                     | Repo ligero; datos reproducibles copiando los ficheros. Confirmado por el usuario.                                                                                                                                                                         |
| 5   | **ECharts** para gráficas, **MapLibre GL** para mapas  | D3 puro, Leaflet                 | ECharts cubre series, heatmaps, treemap y sankey con animaciones de serie; D3 solo si hace falta algo a medida. MapLibre GL renderiza coropletas por WebGL con tooltips fluidos. Se revisará al llegar a la FASE 4.                                        |
| 6   | TypeScript **5.x** en todos los workspaces             | TypeScript 6 (recién publicado)  | Compatibilidad garantizada con typescript-eslint 8; se migrará cuando el ecosistema lo soporte de forma estable.                                                                                                                                           |

## FASE 1 (2026-06-12)

| #   | Decisión                                                                                                                                                                             | Alternativas consideradas             | Motivo                                                                                                                                       |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 7   | Ingesta = **reconstrucción completa** (borra y regenera la BD)                                                                                                                       | Upserts incrementales                 | Idempotencia trivial y garantizada; con ~20k registros el pipeline completo tarda < 1 s.                                                     |
| 8   | Normalización **en SQL dentro de DuckDB** (`read_json` + CTAS)                                                                                                                       | Transformar en TypeScript fila a fila | Set-based, declarativo y rápido; el host solo orquesta. Menos código y menos errores.                                                        |
| 9   | Canonicalización de categorías **data-driven**: variantes agrupadas por clave normalizada (minúsculas, sin tildes, sin puntos finales, espacios colapsados) → variante más frecuente | Listas de sinónimos a mano            | Reproducible sin mantenimiento; el mapa original → limpio queda persistido en `category_mappings` y auditado en el informe de calidad.       |
| 10  | Se conserva `raw_projects` (copia exacta del origen) en la BD                                                                                                                        | Solo la tabla normalizada             | Trazabilidad total a coste despreciable (~6 MB).                                                                                             |
| 11  | Cadenas vacías → `NULL` (PYME, instrumento, área, CNAE…)                                                                                                                             | Categoría «Desconocido»               | NULL expresa ausencia sin inventar categorías; la UI decidirá cómo etiquetarlo.                                                              |
| 12  | `porcentaje_aportacion` en escala **0–100**, `DECIMAL(5,2)`; NULL si presupuesto = 0 (56 filas placeholder)                                                                          | Ratio 0–1                             | Coincide con el filtro slider 0–100 % y evita errores de redondeo flotante.                                                                  |
| 13  | El duplicado lógico detectado (mismo NIF+título+fecha, importes distintos) **se conserva**                                                                                           | Deduplicar                            | Son dos tramos del mismo proyecto; eliminarlo perdería 611 k€ reales. Queda señalado en el informe.                                          |
| 14  | Tabla ordenada físicamente por `fecha_aprobacion` + índices ART en columnas de filtro                                                                                                | Solo índices                          | En DuckDB los zone maps sobre datos ordenados podan rangos de fecha/año; los índices ART ayudan en búsquedas selectivas (NIF, año concreto). |
